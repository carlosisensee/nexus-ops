Arquitetura do Fluxo

Visão completa do sistema — da entrada do cliente até o encerramento do ticket.

Fluxo do Cliente (Portal Público)

1
Identificação
Cliente informa número do pedido ou CPF. Sistema consulta Tiny via API e retorna o pedido.
2
Localização do pedido
Sistema exibe dados do pedido encontrado. Cliente confirma que é o pedido correto.
3
Seleção do produto
Lista de itens do pedido. Cliente seleciona qual produto deseja devolver. Pedidos com múltiplos itens exigem seleção explícita.
4
Motivo da devolução
Seleção de motivo (lista) + campo de observação livre opcional.
5
Instruções de embalagem
Sistema exibe as 8 instruções obrigatórias. Botão de confirmação só fica ativo após scroll até o fim.
6
Confirmação de leitura
Checkbox obrigatório: "Confirmo que li e compreendi as instruções de embalagem."
7
Upload das 3 fotos obrigatórias
Campos separados e rotulados: Ponta 1 · Ponta 2 · Volume completo. Sem todas as 3 não é possível avançar.
✓
Protocolo gerado
Tela de confirmação com número do protocolo (RV-YYYYMMDD-XXXX). Ticket criado automaticamente no sistema interno.

Fluxo Interno (Time de Atendimento)

A
Análise das fotos
Atendente abre o ticket e analisa as 3 fotos. Aprova ou reprova a embalagem.
B
Aprovação e tipo de fluxo
Atendente define: Coleta (transportadora vai até o cliente) ou Postagem (cliente leva à agência).
C
Cotação com transportadora
Sistema gera texto padronizado. Atendente copia e envia à transportadora. Registra retorno (valor, prazo, obs).
D
Confirmação com cliente
Atendente confirma data e hora da coleta (ou gera etiqueta para postagem) e entra em contato com o cliente.
E
Coleta / postagem
Atualização de código de coleta ou voucher. Status avança para "Em trânsito" após confirmação da transportadora.
F
Recebimento e conferência
Expedição registra recebimento, fotografa embalagem antes de abrir e confere produto em até 3 dias úteis.
✓
Finalização
Ticket encerrado. Resultado registrado (aprovado / divergência). Atendente retorna ao cliente.

04

Estrutura do Ticket

Anatomia completa de um ticket de reversa e todos os status possíveis.

Seções do Ticket

1 · Identificação
Número, protocolo, data abertura, status, responsável interno
2 · Dados do Cliente
Nome, CPF, telefone, e-mail, endereço completo (do Tiny)
3 · Dados do Pedido
Número, ID Tiny, NF, data, canal de venda, valor
4 · Produto Devolvido
SKU, nome, qtd, peso, altura, largura, comprimento
5 · Dados da Reversa
Motivo, observação, 3 fotos, tipo de fluxo (coleta/postagem)
6 · Dados da Cotação
Texto gerado, campo de retorno, valor, prazo, observações
7 · Dados da Coleta
Situação, data, confirmação cliente, código/etiqueta/voucher

Pipeline de Status

Protocolo iniciado Aguardando fotos Fotos recebidas Ticket aberto Aguardando aprovação Aprovado p/ cotação Aguardando cotação Cotação pronta Aguard. conf. coleta Coleta solicitada Coleta confirmada Em trânsito Recebido Em conferência Finalizado ✓

Status de Exceção

Embalagem reprovada Pedido não localizado Sem peso/medidas ⚠ Cotação pendente Coleta não realizada
05

Integração com Tiny ERP

Mapeamento completo das chamadas à API do Tiny para enriquecer o ticket.

Endpoints utilizados

GET /pedidos?numero={n}
Busca por número do pedido. Retorna lista resumida.
GET /pedidos?cpf={cpf}
Busca por CPF. Pode retornar múltiplos pedidos — sistema exibe o mais recente com opção de ver outros.
GET /pedido/{id}
Obter pedido completo com itens, cliente, NF e dados logísticos.
GET /produto/{id}
Busca dados logísticos do produto (peso, dimensões). Acionado quando o pedido não inclui essas informações.

Dados extraídos do pedido

→ objeto pedido
numeroNúmero visível no ecommerce
idID interno Tiny (para deep links)
data_pedidoData de criação
nota_fiscalNúmero da NF (se existir)
ecommerce.nomeLojaEcommerceCanal de venda
totalPedidoValor total
→ objeto cliente
nome / cpfIdentificação
fone / emailContato
endereco + cepEndereço de origem da coleta
→ itens[]
sku / descricaoIdentificação do produto
quantidadeQtd devolvida
peso / dimensoesDados para cotação de frete
⚠ Flag automática: Se peso ou qualquer dimensão retornar null/zero no Tiny, o ticket é aberto com status "Produto sem peso/medidas" e aparece destacado em laranja no dashboard, exigindo complemento manual antes da cotação.
06

Regras de Negócio

block
Fotos obrigatórias: O botão "Finalizar solicitação" permanece desabilitado até que os 3 campos de foto (ponta 1, ponta 2, volume) estejam preenchidos. Não há atalho ou exceção.
block
Confirmação de leitura: O checkbox de confirmação das instruções de embalagem é obrigatório. O avanço para upload de fotos só ocorre após marcação.
warning
Produto sem medidas: Se o Tiny não retornar peso ou dimensões, o ticket é criado com flag de exceção e bloqueado para cotação até complemento manual.
warning
Múltiplos pedidos por CPF: Sistema exibe o mais recente com possibilidade de expandir para ver os demais. O cliente deve confirmar qual pedido é o correto.
info
Seleção explícita de item: Mesmo em pedidos de 1 item, o cliente precisa confirmar o produto. Isso previne erros e cria rastreabilidade clara por SKU.
info
Histórico imutável: Toda mudança de status gera um registro em reversa_status_history com timestamp, responsável e observação. Não há edição retroativa.
check_circle
Protocolo único: Formato RV-YYYYMMDD-XXXX. Gerado no backend no momento do create-ticket. Sequencial por dia, não por número global.
check_circle
Texto de cotação automático: Gerado no momento da aprovação, incluindo todos os dados logísticos. Sempre menciona "Conta e Ordem Casa Encanto". Se faltar qualquer dado obrigatório, o campo indica lacuna em [colchetes].
07

Automações

O que acontece automaticamente, sem intervenção manual.

bolt Ao enviar as 3 fotos
  • → Fotos salvas no Supabase Storage (reversa/{protocolo}/)
  • → Ticket criado com status aguardando_aprovacao
  • → Consulta automática ao Tiny pelo número do pedido
  • → Ticket enriquecido com dados do cliente, pedido e produto
  • → Flag de sem medidas ativada se necessário
  • → Protocolo gerado e exibido ao cliente
bolt Ao aprovar o ticket
  • → Status avança para aprovado_cotacao
  • → Texto de cotação montado automaticamente com todos os dados
  • → Botão "Copiar para cotação" ativado no ticket
  • → Registro no histórico de status com timestamp
schedule Alertas de SLA (fase 2)
  • → Ticket sem aprovação após 24h → alerta no dashboard
  • → Cotação não registrada após 2h da aprovação → badge "⚠ Urgente"
  • → Ticket em conferência há mais de 3 dias úteis → sinalização
send Notificações (fase 2)
  • → E-mail de confirmação do protocolo para o cliente
  • → Notificação interna (Slack/e-mail) ao abrir novo ticket
  • → Notificação ao cliente quando coleta for confirmada
  • → Notificação ao cliente após conferência finalizada

Migration SQL

-- 004_reversa.sql
CREATE TABLE reversa_tickets (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  protocolo                 TEXT UNIQUE NOT NULL,
  status                    TEXT NOT NULL DEFAULT 'protocolo_iniciado',
  responsavel               TEXT,
  -- cliente (preenchido via Tiny)
  cliente_nome              TEXT, cliente_cpf TEXT,
  cliente_telefone          TEXT, cliente_email TEXT,
  cliente_endereco          TEXT, cliente_cep TEXT,
  cliente_cidade            TEXT, cliente_uf TEXT,
  -- pedido
  pedido_numero             TEXT, pedido_id_tiny TEXT,
  pedido_data               DATE, pedido_nf TEXT,
  pedido_canal              TEXT, pedido_valor DECIMAL(10,2),
  -- produto devolvido
  produto_sku               TEXT, produto_nome TEXT,
  produto_quantidade        INTEGER DEFAULT 1,
  produto_peso              DECIMAL(8,3),
  produto_altura            DECIMAL(8,2),
  produto_largura           DECIMAL(8,2),
  produto_comprimento       DECIMAL(8,2),
  -- reversa
  motivo                    TEXT,
  observacao_cliente        TEXT,
  foto_ponta1_url           TEXT,
  foto_ponta2_url           TEXT,
  foto_volume_url           TEXT,
  tipo_fluxo                TEXT CHECK (tipo_fluxo IN ('coleta','postagem')),
  -- cotação
  cotacao_texto             TEXT,
  cotacao_retorno           TEXT,
  cotacao_valor             DECIMAL(10,2),
  cotacao_prazo             INTEGER,
  cotacao_obs               TEXT,
  -- coleta
  coleta_status             TEXT,
  coleta_data_solicitada    DATE,
  coleta_confirmacao_cliente BOOLEAN DEFAULT FALSE,
  coleta_codigo             TEXT,
  -- flags
  flag_sem_medidas          BOOLEAN DEFAULT FALSE,
  flag_embalagem_reprovada  BOOLEAN DEFAULT FALSE,
  created_at                TIMESTAMPTZ DEFAULT NOW(),
  updated_at                TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE reversa_status_history (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ticket_id       UUID REFERENCES reversa_tickets(id) ON DELETE CASCADE,
  status_anterior TEXT,
  status_novo     TEXT NOT NULL,
  observacao      TEXT,
  responsavel     TEXT,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE reversa_tickets ENABLE ROW LEVEL SECURITY;
ALTER TABLE reversa_status_history ENABLE ROW LEVEL SECURITY;
