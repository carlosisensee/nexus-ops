Infraestrutura & Token Tiny

O token do Tiny é um recurso global do Nexus OPS — usado em qualquer módulo que precise consultar pedidos, produtos ou clientes.

Padrão adotado — igual ao Supabase

check_circlenetlify/functions/_supabase.js
Já existe. Centraliza a conexão com o banco. Importado por qualquer Function que precisar do DB.
new_releasesnetlify/functions/_tiny.js novo
Criado nesta sprint. Centraliza todas as chamadas ao Tiny ERP. Qualquer módulo futuro (Comercial, Warehouse, etc.) importa e usa diretamente.

Funções exportadas por _tiny.js

searchOrderByNumber(n)Busca por número do pedido
searchOrderByCpf(cpf)Busca por CPF, retorna ordenado por data desc
getOrder(id)Pedido completo por ID interno Tiny
getProduct(id)Produto com peso e dimensões
searchProductBySku(sku)Busca produto por SKU
normalizePedido(p, sku)Normaliza pedido para o formato do ticket
buildCotacaoText(ticket)Gera texto de cotação para transportadora
⚙ Variáveis de ambiente necessárias (Netlify)
TINY_API_TOKEN → Tiny ERP → Configurações → API → Tokens
EMPRESA_CEP_DESTINO → CEP da expedição da empresa
EMPRESA_CIDADE_DESTINO → ex: "São Paulo"
EMPRESA_UF_DESTINO → ex: "SP"
📝 Usage em qualquer Netlify Function
const tiny = require('./_tiny');

// Buscar pedido por número
const pedidos = await tiny.searchOrderByNumber('12345');

// Obter completo e normalizar
const raw = await tiny.getOrder(pedidos[0].id);
const dados = tiny.normalizePedido(raw, skuSelecionado);

✓

Ticket encerrado. Resultado registrado. Cliente notificado.

05

Estrutura do Ticket

7 seções do ticket

1 · Identificação
Número, protocolo, data, status, responsável
2 · Dados do Cliente
Nome, CPF, tel, e-mail, endereço (via Tiny)
3 · Dados do Pedido
Número, ID Tiny, NF, data, canal, valor
4 · Produto Devolvido
SKU, nome, qtd, peso, dimensões
5 · Dados da Reversa
Motivo, obs, 3 fotos, tipo de fluxo
6 · Dados da Cotação
Texto gerado, retorno, valor, prazo
7 · Dados da Coleta
Status, data, confirmação, código/etiqueta

Pipeline de status

Protocolo iniciado Aguardando fotos Fotos recebidas Aguard. aprovação Aprovado p/ cotação Aguard. cotação Cotação pronta Aguard. conf. coleta Coleta solicitada Coleta confirmada Em trânsito Recebido Em conferência Finalizado ✓

Status de exceção

Embalagem reprovada Pedido não localizado Sem peso/medidas ⚠ Cotação pendente Coleta não realizada

Integração Tiny ERP

Todos os dados do cliente, pedido e produto são puxados automaticamente do Tiny ao criar o ticket, via _tiny.js.

Endpoints utilizados
GETpedidos.pesquisa.php?numero=
Busca por número do pedido
GETpedidos.pesquisa.php?pesquisa=CPF
Busca por CPF · retorna lista ordenada
GETpedido.obter.php?id=
Pedido completo: itens, cliente, NF
GETproduto.obter.php?id=
Peso e dimensões (quando ausentes no item)
⚠ Flag automática: Se peso ou qualquer dimensão retornar null/zero, o ticket abre com "Sem peso/medidas" e fica bloqueado para cotação até complemento manual.
Dados extraídos por normalizePedido()
→ Cliente: nome, CPF, tel, email, endereço, CEP, cidade, UF
→ Pedido: número, ID Tiny, NF, data, canal, valor total
→ Produto: SKU, descrição, qtd, peso, dimensões
→ flag_sem_medidas calculada automaticamente
07

Regras de Negócio

block
3 fotos obrigatórias: Botão "Finalizar" desabilitado até ponta1 + ponta2 + volume preenchidos. Sem exceção.
block
Confirmação de leitura obrigatória: Checkbox das instruções de embalagem é pré-requisito para avançar ao upload de fotos.
warning
Sem medidas: Flag automática se Tiny não retornar peso ou dimensões. Ticket sinalizado; cotação bloqueada até complemento.
warning
Múltiplos pedidos por CPF: Exibe o mais recente por padrão. Cliente deve confirmar qual pedido é o correto antes de avançar.
info
Seleção explícita de item: Mesmo em pedidos de 1 item. Cria rastreabilidade por SKU e previne erro de devolução do produto errado.
info
Histórico imutável: Toda mudança de status gera registro em reversa_status_history com timestamp e responsável.
check_circle
Protocolo único por dia: Formato RV-YYYYMMDD-XXXX. Sequencial diário. Gerado no backend.
check_circle
Texto de cotação automático: Gerado na aprovação com todos os dados. Campos ausentes aparecem como [A INFORMAR].

Automações

boltAo enviar as 3 fotos
  • → Upload das fotos no Supabase Storage
  • → Ticket criado — status aguardando_aprovacao
  • → Consulta automática ao Tiny pelo pedido
  • → normalizePedido() enriquece o ticket
  • → Flag sem_medidas calculada
  • → Protocolo gerado e exibido ao cliente
boltAo aprovar o ticket
  • → Status → aprovado_cotacao
  • → buildCotacaoText() gera o texto completo
  • → Botão "Copiar para cotação" ativado
  • → Registro no histórico de status
scheduleAlertas de SLA (fase 2)
  • → Sem aprovação em 24h → badge "Urgente"
  • → Cotação não registrada em 2h → alerta
  • → Em conferência há +3 dias → sinalização
sendNotificações (fase 2)
  • → E-mail ao cliente com protocolo
  • → Notificação interna ao abrir ticket
  • → Notificação ao cliente na confirmação da coleta

CREATE TABLE reversa_tickets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  protocolo TEXT UNIQUE NOT NULL,
  status TEXT NOT NULL DEFAULT 'protocolo_iniciado',
  responsavel TEXT,
  cliente_nome TEXT, cliente_cpf TEXT, cliente_telefone TEXT,
  cliente_email TEXT, cliente_endereco TEXT, cliente_cep TEXT,
  cliente_cidade TEXT, cliente_uf TEXT,
  pedido_numero TEXT, pedido_id_tiny TEXT, pedido_data DATE,
  pedido_nf TEXT, pedido_canal TEXT, pedido_valor DECIMAL(10,2),
  produto_sku TEXT, produto_nome TEXT, produto_quantidade INTEGER DEFAULT 1,
  produto_peso DECIMAL(8,3), produto_altura DECIMAL(8,2),
  produto_largura DECIMAL(8,2), produto_comprimento DECIMAL(8,2),
  motivo TEXT, observacao_cliente TEXT,
  foto_ponta1_url TEXT, foto_ponta2_url TEXT, foto_volume_url TEXT,
  tipo_fluxo TEXT CHECK (tipo_fluxo IN ('coleta','postagem')),
  cotacao_texto TEXT, cotacao_retorno TEXT,
  cotacao_valor DECIMAL(10,2), cotacao_prazo INTEGER, cotacao_obs TEXT,
  coleta_status TEXT, coleta_data_solicitada DATE,
  coleta_confirmacao_cliente BOOLEAN DEFAULT FALSE, coleta_codigo TEXT,
  flag_sem_medidas BOOLEAN DEFAULT FALSE,
  flag_embalagem_reprovada BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE TABLE reversa_status_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ticket_id UUID REFERENCES reversa_tickets(id) ON DELETE CASCADE,
  status_anterior TEXT, status_novo TEXT NOT NULL,
  observacao TEXT, responsavel TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
ALTER TABLE reversa_tickets ENABLE ROW LEVEL SECURITY;
ALTER TABLE reversa_status_history ENABLE ROW LEVEL SECURITY;
